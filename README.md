# Multi_Memory_chatbot

一个正在持续更新中的多角色长期记忆聊天系统原型。

它不是一个普通的 chatbot，而是一个结合了“多角色群聊”“长期记忆”“线程摘要”“子梦系统”的实验性 AI 对话项目。

[项目效果图](./会话效果展示.png)

## 项目简介

这个项目的目标，是让用户不只是和单一 AI 对话，而是可以在同一个系统中：

- 召唤不同角色进行交流
- 让多个角色在同一上下文中共同参与讨论
- 让系统记录会话过程并生成线程摘要
- 让系统从摘要中提取有价值的信息，沉淀到长期记忆中
- 存储系统多角色通用

当前项目仍处于早期版本，整体功能已经具备基础雏形，但工程结构、文档说明和细节体验还在继续完善。

## 核心功能

### 1. 多角色沟通系统

用户可以自定义加入自己喜欢的角色，例如来自：

- 影视剧
- 游戏
- 小说

在聊天时，用户可以：

- 选择某个角色作为当前对话角色
- 在发言时直接点名某个角色

系统内部为角色设置了统一的群聊式 Prompt 规则，使角色能够理解：

- 自己并不是在进行一对一对话
- 当前处在一个多人对话环境中
- 历史消息中不同的话分别是谁说的
- 可以对其他角色的观点做出回应、补充或点评

目前项目默认保留两个内置角色：

- 英剧人物 汉弗莱（`humphrey`）
- 只狼NPC，永真（`yongzhen`）

### 2. 长期记忆系统

系统包含中期记忆与长期记忆两层数据结构。

这些记忆文件用于记录例如：

- 用户画像
- 用户偏好
- 用户观点
- 用户关系描述
- 会话中提取出的重要事件

在新会话或后续对话中，系统可以根据语义召回相关记忆，并将其导入上下文，使角色更了解用户、让回应更连续。

### 3. 子梦系统

“子梦系统”是这个项目的重要特色之一。

它基于会话存储系统工作，而会话存储主要由两部分组成：

- 实时会话记录（raw）
- 当前线程的摘要（summary）

当用户选中某个线程并执行导入时，系统会把该线程的摘要信息交给大模型，通过内置 Prompt 判断：

- 哪些信息有价值
- 哪些内容适合存入长期记忆系统
- 应该写入哪类记忆文件

随后系统将自动执行记忆沉淀。

## 当前目录结构

```text
Multi_Memory_chatbot/
├─ cmd/
│  ├─ launcher/              # 一键启动入口（启动后端与前端静态服务）
│  └─ whale/                 # CLI / HTTP 服务入口
├─ data/
│  ├─ memory/
│  │  ├─ digest/             # 最近讨论摘要
│  │  ├─ index/              # 记忆索引
│  │  ├─ long_term/          # 长期记忆（画像、关系、偏好、观点）
│  │  └─ mid_term/           # 中期记忆（事件、情绪等）
│  ├─ sessions/
│  │  ├─ roles.json          # 角色列表
│  │  ├─ thread_states.json  # 线程状态
│  │  └─ windows.json        # 会话窗口列表
│  └─ system/
│     ├─ llm_config.json     # 模型配置
│     ├─ memory_policy.json  # 记忆策略
│     └─ runtime_state.json  # 运行时状态
├─ frontend/                 # 前端页面与脚本
├─ internal/
│  ├─ api/                   # HTTP API
│  ├─ config/                # 配置加载
│  ├─ llm/                   # 大模型调用封装
│  ├─ memory/                # 记忆处理逻辑
│  ├─ runtime/               # 核心运行时与角色系统
│  ├─ session/               # 会话、窗口、raw/summary 管理
│  └─ tools/                 # 工具层
├─ index.html
├─ go.mod
└─ README.md
```

## 技术栈

当前项目以 Go 为主，配合原生前端页面构成一个轻量原型：

- Go 1.22
- 原生 HTML / CSS / JavaScript
- 本地 JSON 文件作为会话与记忆存储

## 运行方式

### 方式一：使用 launcher 启动

在项目根目录执行：

```bash
go run ./cmd/launcher
```

默认行为：

- 启动后端服务：`http://127.0.0.1:8080`
- 启动前端静态服务：`http://127.0.0.1:5173/frontend/`

说明：

- `launcher` 会尝试清理端口占用
- 当前实现对 Windows 环境做了专门处理
- 前端静态服务依赖本机可用的 `python`

### 方式二：仅启动后端

```bash
go run ./cmd/whale serve
```

### 方式三：CLI 模式

```bash
go run ./cmd/whale
```

CLI 中目前支持的基础命令包括：

- `/new [标题]`：新建窗口
- `/window window-001`：切换到已有窗口
- `/list`：查看窗口列表
- `/dream`：手动触发子梦系统

## 配置说明

模型配置文件位于：

- `data/system/llm_config.json`

你需要根据自己的模型服务填写：

- `provider`
- `base_url`
- `api_key`
- `model`

请注意：

- 不要把真实 `api_key` 上传到 GitHub
- 建议后续改造为 `.env` 或本地私有配置方式

## 数据与隐私说明

本项目包含会话、摘要、记忆等数据结构。

在公开仓库前，建议不要提交以下内容中的真实数据：

- 原始会话记录
- 自动摘要结果
- 用户画像与长期记忆
- API Key
- 本地测试数据

当前仓库更适合作为：

- 原型展示
- 架构说明
- 功能思路演示

而不是直接承载真实用户数据。
